# 用户体验分析：素材 → 速推生产视频 → 发布到账号

基于「用素材ID xxx 调用速推生产一个视频，并发布到 NICK」的完整流程（浏览器登录后台、在会话里发送），结合前端与后端代码与日志行为做的体验分析。

---

## 1. 流程概览（用户视角）

| 阶段 | 用户看到的主文案 | 步骤列表（逐条追加/更新） |
|------|------------------|---------------------------|
| 发消息后 | 正在思考... | （无） |
| 查素材/账号/登录 | 正在思考... | ✓ 查看素材 完成 → ✓ 查看账号 完成 → ✓ 检查登录 完成 |
| 提交视频任务 | 正在思考... | 已提交视频生成任务… → ✓ 任务已提交，正在生成视频（约 1–3 分钟）… |
| 轮询生成结果 | 视频生成中，请稍候… | 正在获取生成结果（约 1–3 分钟）… 请保持本页打开。 |
| 轮询中（每 15s） | 正在查询生成结果…（15秒）/（30秒）/… | 最后一行保持「正在查询…」或「✓ 视频已生成」 |
| 生成完成 | 视频生成中，请稍候… | ✓ 视频已生成 → **新增** 正在生成回复… |
| 发布 | 正在思考... | 正在 发布内容… → ✓ 发布内容 完成 |
| 结束 | （气泡消失） | 助手回复逐行打出 |

---

## 2. 已实现的体验优点

- **进度可见**：每个工具调用都有「正在 xxx…」→「✓ xxx 完成」，用户知道在查素材、查账号、提交任务、发布等。
- **长时间等待有反馈**：轮询阶段每 15 秒下发 `task_poll`，主文案会变成「正在查询生成结果…（15秒）」「（30秒）」等，避免「卡住」感。
- **预期管理**：步骤里明确写「约 1–3 分钟」「请保持本页打开」，降低用户中途离开或刷新。
- **状态与回复分离**：`status: "正在生成回复…"` 改为**追加**一条新步骤（已改），不再覆盖「✓ 视频已生成」，结果与后续动作更清晰。

---

## 3. 已修复的体验问题

- **「✓ 视频已生成」被覆盖**：原先 `status: "正在生成回复…"` 用 `updateLastChatTypingStep` 覆盖最后一行，导致「✓ 视频已生成」消失。现改为 `appendChatTypingStep`，保留「✓ 视频已生成」并新增「正在生成回复…」。

---

## 4. 可继续优化的点

1. **前段主文案过泛**  
   查素材/账号/登录、发布时主文案仍是「正在思考...」。可考虑在这些工具阶段改为更贴切的文案（如「正在查询账号与素材…」「正在发布…」），需后端在对应 `tool_start` 时带 `status` 或前端根据 `ev.name` 映射主文案。

2. **轮询阶段主文案**  
   当前轮询时主文案为「正在查询生成结果…（N秒）」，已足够清晰。若希望更统一，可与步骤里「正在获取生成结果…」的表述略作统一（可选）。

3. **无素材附件时的说明**  
   若用户**只输入文字**「用素材ID xxx…」而**未在输入框上方上传/选择图片**，后端不会注入素材 URL，模型可能报「素材不存在」或无法图生视频。产品上可在输入区或帮助文案中提示：图生视频需先上传图片或选择已有素材。

4. **错误态在步骤中的呈现**  
   工具失败时步骤仍显示「✓ xxx 完成」，真实失败原因在最终助手回复里。若要在步骤里区分成功/失败，需后端在 `tool_end` 中增加 `ok: false` 或 `error` 字段，前端对失败显示「✗ xxx：原因」等。

---

## 5. 日志定位（排查用）

在 **logs/app.log** 或页面「日志」Tab 中可按以下关键字串联整条链路：

| 关键字 | 含义 |
|--------|------|
| `[CHAT] 注入素材 URL` | 本条消息带 attachment_asset_ids 时，注入的素材 URL |
| `[CHAT] tool_call:` | 模型发起的工具调用（list_assets / invoke_capability / publish_content 等） |
| `[CHAT] task.get_result 轮询` | 速推任务轮询，每 15s 一条 |
| `[MCP] invoke_capability` | MCP 侧能力调用（含 video.generate / task.get_result） |
| `[MCP] publish_content 调用` / `publish_content 后端响应` | 发布请求与后端返回 |
| `[PUBLISH-API]` | 发布接口收到请求、执行、结果 |

---

## 6. 小结

- 流程已跑通：浏览器登录 → 会话发消息 → 工具步骤与轮询进度可见，发布有明确步骤与结果。
- 本次修改保证了「✓ 视频已生成」不会被「正在生成回复…」覆盖，步骤列表更符合实际状态顺序。
- 后续可按「可继续优化的点」做文案与错误态的小步优化，并配合日志快速排查问题。
