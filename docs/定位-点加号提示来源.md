# 定位：「点加号、发布页面准备好后告诉我」这句话从哪来

## 结论（先看这个）

- **代码里没有任何地方写死这句话。**
- 可能来源只有两个：
  1. **模型自己推出来的**：根据工具说明里的「会激活到最前面」「发布前需要登录」等，结合对抖音的常识，自己编了「点加号、到发布页、准备好后告诉我」。
  2. **对话历史里已经出现过**：之前某轮回复里模型说过这句话，被存进 history，后面几轮模型一直能看到，就会反复说或延续。

下面是怎么一步步定位的。

---

## 1. 模型生成那句回复时，能看到哪些输入？

模型在生成「点加号、发布页面准备好后告诉我」这类回复时，输入只有这几类：

| 输入来源 | 文件位置 | 是否包含「点加号/发布页面准备好」 |
|----------|----------|----------------------------------|
| 系统提示 system prompt | `backend/app/api/chat.py` 约 703–736（/chat）、850–884（/chat/stream） | **否**。已全文搜索，无该句。 |
| 工具定义 tools | `mcp/http_server.py` 约 227–274，通过 MCP 传给模型 | **否**。无「点加号」「准备好后告诉我」。只有「会激活到最前面」「发布前需要登录」「确保账号已登录」。 |
| 工具返回 tool result | 来自后端 API，MCP 把 JSON 转成字符串塞进对话 | **否**。见下一节。 |
| 对话历史 history | 前端传的 `payload.history`，可能是之前多轮 user/assistant | **不确定**。若之前某条 assistant 消息里已经出现过这句话，就会一直在 history 里。 |

所以：**代码里没有任何一段文字直接等于「点加号、发布页面准备好后告诉我」。**

---

## 2. 工具返回里到底有什么？

- **open_account_browser**  
  后端：`publish.py` 的 `open_browser` 返回  
  `{"ok": True, "logged_in": True/False, "message": "浏览器已打开，当前已登录" 或 "浏览器已打开，请扫码登录"}`  
  → 没有「加号」「发布页面」「准备好后告诉我」。

- **publish_content**  
  后端：`publish.py` 的 `create_publish_task` 返回  
  `{"task_id": ..., "status": "success"|"failed"|"need_login", "result_url": ..., "error": ...}`  
  未登录时错误文案是：「未登录，已打开浏览器登录页，请扫码登录后再重试发布」。  
  → 同样没有「点加号」「发布页面」「准备好后告诉我」。

所以：**工具返回里也没有这句话。**

---

## 3. 工具描述里可能被「联想」成那句话的部分

在 `mcp/http_server.py` 里和发布/浏览器相关的只有：

- `open_account_browser`：  
  「打开指定账号的浏览器窗口（**会激活到最前面**）。如果未登录会显示登录页。用于：用户要求打开某账号浏览器、**发布前需要登录**等场景。」
- `publish_content`：  
  「将本地素材发布到指定平台账号（如抖音、B站）。需先通过 save_asset 保存素材，**并确保账号已登录**。」

模型可能做的推理是：

- 看到「会激活到最前面」→ 想到要提醒用户「把窗口切到最前面」；
- 看到「发布前需要登录」「确保账号已登录」+ 平台是「抖音」→ 从通用知识里联想到「抖音发布要进发布页」「发布页经常是点加号进去」；
- 再自己编出「到发布页面准备好后告诉我」。

所以：**这句话不是从某一行代码复制出来的，而是模型根据工具描述 + 常识自己生成的。**

---

## 4. 怎么验证是「模型推断」还是「历史里已有」？

**方法 A：新会话复现**

- 开一个**全新对话**（不要带历史）。
- 直接说「帮我把 xxx 发到抖音」或先打开浏览器再发布。
- 若**新会话里仍然出现**「点加号、发布页面准备好后告诉我」→ 基本可以认定是**模型根据工具说明 + 常识推断**出来的。
- 若**新会话里不再出现**，只有老会话里才有 → 很可能是**历史里曾经出现过**，后面几轮只是在延续/重复。

**方法 B：看发往模型的 messages**

- 在 `backend/app/api/chat.py` 里，在调用 `_chat_anthropic` / `_chat_openai` 之前，把 `messages` 打日志（或临时 return 出来）：
  - 看最后一条 user 消息之前，是否已经有一条 assistant 消息里包含「点加号」「发布页面」「准备好后告诉我」。
- 若**有** → 来源就是**对话历史**（之前某轮模型自己说过，被存进 history）。
- 若**没有** → 来源就是**当前轮模型新生成的**，也就是模型推断。

---

## 5. 总结：哪里出了问题、怎么定位

- **「哪里出了问题」**  
  - 不是某段写死的业务逻辑「错」了，而是**模型在「打开浏览器 / 发布到抖音」这类场景下，会自己生成这句提示**。
  - 触发材料来自：工具描述里的「会激活到最前面」「发布前需要登录」「确保账号已登录」+ 模型对抖音发布流程的常识。

- **怎么定位**  
  1. 用**新会话**试一次（方法 A），区分是「每次都会说」还是「只有带历史的会话才说」。  
  2. 用**打印/查看 messages**（方法 B），确认这句话是已经在 history 里，还是当轮新生成的。  
  3. 确认后，若需要避免这句话，再考虑在系统提示里加一句约束（例如：发布流程全自动完成，不要要求用户手动点加号或说「准备好」）。  

本文档只做定位与复现方法，不包含任何代码修改建议。
